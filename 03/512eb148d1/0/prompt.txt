### FINDING 1 — CRITICAL: Empty catch block swallows body parse errors without logging

**Location:** `/Users/jonassilva/Documents/developer/maquina-de-conteudo/.claude/worktrees/fase-6-impl/src/app/api/themes/[id]/wizard/route.ts` line 17

**Code:**
```typescript
try { body = await request.json() } catch { /* optional body */ }
```

**Issue:** The catch block is completely empty with only a comment. This is explicitly forbidden by the project's own rules ("Empty catch blocks are never acceptable"). While the intent is to treat the body as optional, the current implementation will silently swallow any error — including errors completely unrelated to an empty body, such as a `SyntaxError` caused by malformed JSON sent by the client (e.g., `{"motor": tribal_v4}` — missing quotes). The caller then silently falls through to the default motor `"tribal_v4"` with zero indication that their request body was ignored.

**Hidden errors this catch block suppresses:**
- `SyntaxError` from malformed JSON (client bug that should be surfaced as a 400)
- `TypeError` from a network stream read failure
- Any future error thrown inside `request.json()` due to middleware or platform behavior

**User impact:** A client sending `motor: "brandsdecoded_v4"` with a malformed JSON body will get a Tribal wizard instead of a BrandsDecoded wizard, with no error returned. The user ends up in the wrong wizard and has no idea why.

**Recommendation:** At minimum, log the body parse failure and, for malformed JSON (`SyntaxError`), return a 400 instead of falling back silently. For a missing body, allow the fallback, but make it explicit.

```typescript
let body: { motor?: Motor } = {}
try {
  body = await request.json()
} catch (err) {
  if (err instanceof SyntaxError) {
    return NextResponse.json({ error: "Invalid JSON in request body" }, { status: 400 })
  }
  // Body is optional — absence is allowed, but log unexpected parse failures
  console.warn("[wizard/route] Could not parse request body, using defaults:", err)
}
```

### FINDING 2 — CRITICAL: Raw internal error message leaked to the client

**Location:** `/Users/jonassilva/Documents/developer/maquina-de-conteudo/.claude/worktrees/fase-6-impl/src/app/api/themes/[id]/wizard/route.ts` lines 27–30

**Code:**
```typescript
} catch (err) {
  const msg = err instanceof Error ? err.message : String(err)
  return NextResponse.json({ error: msg }, { status: 500 })
}
```

**Issue:** The outer catch leaks the raw `err.message` string directly into the HTTP response body. `createWizardFromTheme` performs database operations — if Drizzle, Neon, or the PostgreSQL driver throws, the message could contain:
- SQL query fragments
- Table or column names
- Connection strings or DSN components
- Internal schema details (e.g., `duplicate key value violates unique constraint "content_wizards_pkey"`)

These are internal implementation details that expose the database schema to any client making the request. This violates security best practices and makes the API inconsistent with the project's own `AppError`/`toAppError` pattern defined in `src/lib/errors.ts`.

Additionally, there is no `console.error` call, so server-side logs will have no record of this failure. A 500 error that is not logged is impossible to diagnose.

**Recommendation:** Use the project's established `toAppError` helper, log the full error server-side, and return a sanitized, generic message for unexpected errors. The `theme not found` case (a predictable `Error` thrown by `createWizardFromTheme`) should be mapped to a 404 specifically.

```typescript
import { toAppError, isAppError, NotFoundError } from "@/lib/errors"

} catch (err) {
  const appError = toAppError(err, "WIZARD_CREATE_FAILED")
  console.error("[wizard/route] Failed to create wizard for theme", themeId, appError)

  if (appError.message.includes("not found") || appError instanceof NotFoundError) {
    return NextResponse.json({ error: `Theme ${themeId} not found` }, { status: 404 })
  }

  // Never expose internal error details to the client
  return NextResponse.json(
    { error: "Failed to create wizard. Please try again." },
    { status: appError.statusCode ?? 500 }
  )
}
```

The ideal fix is to have `createWizardFromTheme` throw a `NotFoundError` from `src/lib/errors.ts` instead of a generic `new Error(...)` on line 51 of `create-wizard.ts`, which would allow the route to identify the error class precisely rather than doing string matching.
