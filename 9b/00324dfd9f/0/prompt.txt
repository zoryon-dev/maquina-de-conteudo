It contains non-trivial logic: auth guard, wizard ownership check, a check for existing `generatedContent`, a lookup of a specific headline by `headlineId` within the previous result, and a full orchestrator re-invocation. The failure modes unique to this action — headline not found in prior result, no prior generation existing, and the re-run persisting correctly — are entirely unverified.

Suggested additions:
- auth null → returns `{ success: false, error: /auth/ }`
- `wiz.generatedContent` is null/undefined → returns `{ success: false, error: /nenhuma geração anterior/ }`
- `headlineId` does not match any headline in `prev.headlines` → returns `{ success: false, error: /headline não encontrada/ }`
- Success path: orchestrator called with `forcedHeadlineId` set and `autoSelectHeadline: false`

**2. DB mock ignores `where` clause — ownership check is untested (criticality: 8)**

In the test file, the mock is:

where: () =&gt; ({

The `where` function is a no-op that always returns whatever is in `wizardStateRef.current`, completely ignoring the `userId` argument. A user accessing another user's wizard would see `undefined` from the real DB, but the mock always returns the fixture regardless.

The existing "wizard not found" test (where `wizardStateRef.current = undefined`) does exercise the not-found return path, but only because it explicitly sets the ref to `undefined` — not because the mock filters by userId. it("retorna erro quando wizard pertence a outro usuário", async () =&gt; {

**4. RAG failure graceful degradation test in `generateBdContentAction` (criticality: 6)**

## Test Quality Issues

**8. DB mock ignores `where` arguments — tautological ownership tests**

As described in gap #2, the `where` mock is a passthrough. Every "not found" test works not because the DB enforces ownership but because the test manually sets `wizardStateRef.current = undefined`. The store exports `setGeneratedResult` (which sets `generatedResult` on state), but no test covers it. The `reset` test verifies `generatedResult` is cleared (implicitly, since `initialState` has it as `undefined`), but there is no test that `setGeneratedResult(r)` actually persists `r` and is independent of `setSelectedHeadlineId`.

It does not check the inverse question: shared `@/lib/` modules that should be accessed only via `@/lib/` are correctly accessed. This is acceptable — the isolation contract is about not crossing the app directory boundary, and `@/lib/` is explicitly shared. The test is logically sound and non-tautological: it reads actual filesystem contents and would catch a real violation. The sanity assertions (`bdFiles.length &gt; 0`, `tribalFiles.length &gt; 0`) are good practice.

One blind spot: `@/stores/bd-wizard-store` is imported from the BD wizard page (`page.tsx`) and components. The isolation test's `forbiddenAbsolute` patterns do not block imports from `@/stores/`, so if Tribal components somehow imported from `@/stores/bd-wizard-store`, it would not be caught. This is a minor gap since the Tribal wizard uses its own separate store, but it is not enforced.
